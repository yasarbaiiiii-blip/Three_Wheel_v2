# benchmark_rpp — Baseline & Regression Capture Procedure

This is the **one** procedure for every cross-track baseline run on the 3WD
rover. Run it before any RPP upgrade to capture a "before" number; run it
again after the upgrade to prove the delta. Without a baseline you cannot
prove an improvement — every task in `task_rpp_upgrade/` depends on this.

## 0. One-time setup (on the analysis machine)

```bash
pip install rosbags pandas matplotlib numpy
```

The `rosbags` library is pure Python — no rclpy / ROS install required. It
reads both `mcap` and sqlite3 (`db3`) rosbag2 formats on Linux **and**
Windows, so you can analyse on the laptop without going to the Jetson.

## 1. Pre-flight (rover, before every run)

1. Power on. Wait for the FCU to boot and `px4-dxp.service` to be active.
2. Verify RTK FIXED: `ros2 topic echo /mavros/gpsstatus/gps1/raw --once` —
   `fix_type` must be `6` (RTK fixed). HDOP must be < 0.7.
3. Verify the param set loaded on the FCU matches the one you intend to
   benchmark (e.g. `Final_Best`). Read it back from QGC and screenshot.
4. Place the rover at the **start point** of the path:
   - Square: south-west corner facing North.
   - Half-circle: tangent point at start heading.
   - Straight 5 m: at the southern end facing North.
5. Confirm `/rpp/debug` is publishing:
   `ros2 topic hz /rpp/debug` — expect ~50 Hz when controller is running.
   If silent, `rpp_controller_node` is not running — fix that before recording.

## 2. Record the bag (Jetson)

In one terminal:

```bash
cd ~/PX4_DXP
mkdir -p baseline/$(date +%F)_<path>_<params>/bag
cd baseline/$(date +%F)_<path>_<params>/bag

ros2 bag record \
  /rpp/debug \
  /mavros/local_position/pose \
  /mavros/global_position/global \
  /mavros/state \
  /mavros/setpoint_raw/local \
  /rpp/velocity_ned \
  -o run
```

In a second terminal, trigger the mission via the FastAPI server:

```bash
# e.g. via curl, or the rover-server UI
curl -X POST http://192.168.1.102:8000/mission/run \
  -H 'Content-Type: application/json' \
  -d '{"mission": "Test_mission/mission_square.waypoints"}'
```

**Stop the bag** with `Ctrl-C` once the controller reaches `DONE`
(`/rpp/debug[7] == 3`). Sit through 1-2 s of zero-velocity output to make
the endpoint sample unambiguous.

## 3. Analyse

```bash
python tools/benchmark_rpp.py \
  --bag    baseline/2026-05-24_square_Final_Best/bag/run \
  --path   square \
  --params Final_Best \
  --mission Test_mission/mission_square.waypoints \
  --out    baseline/2026-05-24_square_Final_Best
```

This produces:

```
baseline/2026-05-24_square_Final_Best/
  bag/                    # rosbag (gitignored)
  debug.csv               # decoded per-sample
  summary.txt             # one-line summary
  xtrack.png              # xtrack vs time + xtrack vs arclength
  notes.md                # YOU write this — surface, weather, anomalies
```

The summary line uses this **canonical format** (do not edit by hand):

```
path=square | n=1234 | t=23.4s | xtrack max=0.094 mean=0.011 rms=0.036 | heading_err rms_deg=2.1 | endpoint_err_m=0.018
```

Field meanings:

| Field | Meaning |
|---|---|
| `n` | number of `/rpp/debug` samples in the run |
| `t` | wall-clock duration of the run, seconds |
| `xtrack max` | peak `|xtrack|` over the run, metres |
| `xtrack mean` | mean `|xtrack|`, metres |
| `xtrack rms` | RMS of signed xtrack, metres |
| `heading_err rms_deg` | RMS of `/rpp/debug[1]`, converted to degrees |
| `endpoint_err_m` | great-circle distance from final GPS fix to last mission WP. `n/a` if `--mission` not passed or `/mavros/global_position/global` not in bag |

## 4. Standard baseline directory layout

```
baseline/
  2026-05-24_square_Final_Best/
    bag/                  # rosbag — GITIGNORED, never committed
    debug.csv             # decoded
    summary.txt           # one-line summary
    xtrack.png            # plot
    notes.md              # human notes (surface, wind, anomalies, RTK status)
```

Naming convention: `YYYY-MM-DD_<path>_<params>[<_suffix>]/`. The suffix
distinguishes A/B runs (`_run2`, `_with_feature07`, …).

`baseline/*/bag/` is gitignored. Commit everything else.

## 5. The three baselines you must capture before any RPP upgrade

The user runs these once on the rover. The bag is captured per § 2, the
summary per § 3.

| # | Path | Mission file | Params |
|---|---|---|---|
| 1 | 2 m square | `Test_mission/mission_square.waypoints` | `Final_Best` |
| 2 | Half circle (180°, Karney) | `Test_mission/mission_half_circle_180.waypoints` | `Final_Best` |
| 3 | Straight 5 m | `Test_mission/mission_straight_5m.waypoints` | `Final_Best` |

The straight-5m mission was generated by this task; the other two already
exist in `Test_mission/`.

Once captured, commit the three `summary.txt` + `debug.csv` + `xtrack.png`
+ `notes.md` sets — they become the numbers every future upgrade must beat.

## 6. Troubleshooting

| Symptom | Likely cause | Fix |
|---|---|---|
| `bag has no /rpp/debug` | `rpp_controller_node` was not running during recording | Start `rpp-pipeline.service`, verify `ros2 topic hz /rpp/debug`, re-record |
| All `xtrack` are NaN | `rpp_controller_node` was in `IDLE` (no path loaded) | Trigger the mission *after* `ros2 bag record` starts |
| `endpoint_err_m=n/a` | `--mission` not passed, or `/mavros/global_position/global` missing from bag | Add `--mission <file>` and ensure global topic is in the record list |
| `state_code` stays `4` (RTK_WAIT) | GPS fix < RTK_FIXED | Wait for RTK fix before triggering mission |
| Self-test fails | Logic bug in tool — file an issue, do not paper over | `python tools/benchmark_rpp.py --self-test` |

## 7. Self-test

```bash
python tools/benchmark_rpp.py --self-test
```

Builds synthetic data, exercises the parser + summariser, and prints a
canonical line. Use this on any new analysis machine to confirm the tool
imports + decodes correctly without a real bag.

## 8. Frame discipline (cheat-sheet)

- `/mavros/local_position/pose` is ENU. The tool swaps to NED on read
  (`pos_n = ENU.y`, `pos_e = ENU.x`).
- `/rpp/debug[0]` (`xtrack`) is signed: **+ = right of path** (when looking
  along the path direction). A persistent positive bias means the rover is
  consistently riding right of the intended line.
- All metrics in the summary line are in metres / degrees / seconds — never
  centimetres. Convert at the eyeball stage, not in the data.
